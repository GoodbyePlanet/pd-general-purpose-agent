# PD General Purpose Agent - V2.1 Implementation Plan

## Context

V2.0 introduced web tools (`web_search`, `fetch_url`) directly on a single ReAct agent. V2.1 refactors to a **multi-agent architecture**: a root agent handles delegation and Slack responses, while a dedicated web research subagent owns all internet-facing tools. This separation makes it easy to add future subagents (Coda, GitHub, logs) without bloating the root agent's context.

## Architecture

```
Root Agent  (tools: [web_research])
  └── web_research @tool
        └── Web Research Agent  (tools: [fetch_url, web_search?])
```

**Pattern:** Agent-as-tool — the web research agent is wrapped as a LangChain `@tool`. The root agent calls it when internet access is needed; otherwise it responds directly. No new packages required.

**Files changed:**
- `app/agent.py` — full refactor of internal structure (sole file changing)
- `pyproject.toml` — version bump `2.0.0` → `2.1.0`

**Files unchanged:** `slack_app.py`, `config.py`, `api.py`, `main.py`, `Dockerfile`, `docker-compose.yml`

## Implementation Steps

### Step 1: Version Bump
- Update `pyproject.toml`: `version = "2.1.0"`
- `_read_version()` in `agent.py` picks it up automatically — no other change needed
- **Verify**: `_VERSION` in agent context shows `2.1.0`

### Step 2: New System Prompts
Replace single `SYSTEM_PROMPT` with two prompts:

**`WEB_RESEARCH_SYSTEM_PROMPT`** — task-focused, no personality:
- Always use tools; never answer from memory
- If given a URL → `fetch_url`; if given a topic → `web_search`
- Return raw, factual content (root agent handles formatting)

**`ROOT_SYSTEM_PROMPT`** — full Pidi personality (preserve existing tone and wording):
- References `web_research` tool (not `web_search`/`fetch_url` directly)
- Instructs to use `web_research` for anything internet-related
- Keeps "big things coming" section and Slack markdown guidance

### Step 3: Rename `_build_tools` → `_build_research_tools`
Same logic as before — conditionally includes `web_search` if Tavily key is set. Rename makes intent clear (tools for the research subagent, not the root).

### Step 4: Create Web Research Agent
```python
_web_research_agent = create_react_agent(
    model=_llm,
    tools=_build_research_tools(),
    prompt=WEB_RESEARCH_SYSTEM_PROMPT,
)
```
Must be defined **before** the `web_research` tool that wraps it.

### Step 5: Create `web_research` Tool (wraps sub-agent)
```python
@tool
async def web_research(task: str) -> str:
    """Delegate a web research task to the web research sub-agent.
    Use for: searching the web, fetching URLs, current events, documentation.
    Pass a plain-language description of what you need.
    """
    logger.info("[web-research-agent] Invoked: task=%r", task)
    try:
        result = await _web_research_agent.ainvoke({"messages": [("user", task)]})
        response = result["messages"][-1].content
        tool_calls = sum(1 for m in result["messages"] if getattr(m, "type", None) == "tool")
        logger.info("[web-research-agent] Done: tool_calls=%d, chars=%d", tool_calls, len(response))
        return response
    except Exception as e:
        logger.exception("[web-research-agent] Failed: task=%r", task)
        return f"Web research failed: {e}. Please inform the user the web research tool encountered an error."
```
- Must be `async def` (calls `ainvoke`)
- Error returns string (not raise) — prevents LangGraph from looping on tool errors

### Step 6: Create Root Agent
```python
_root_agent = create_react_agent(
    model=_llm,
    tools=[web_research],
    prompt=ROOT_SYSTEM_PROMPT,
)
```
Must be defined **after** the `web_research` tool.

### Step 7: Update `get_response` Logging
Same signature and logic; update log prefixes for clarity:
```python
logger.info("[root-agent] Invoking with %d message(s)", len(messages))
logger.info("[root-agent] Finished: tool_calls=%d, response_chars=%d", ...)
```

## Module-Level Construction Order

Critical — Python executes top-to-bottom:

1. `_read_version()`, `_VERSION`
2. `WEB_RESEARCH_SYSTEM_PROMPT`, `ROOT_SYSTEM_PROMPT`
3. `_tavily_client`, `_llm` (unchanged)
4. `@tool fetch_url`, `@tool web_search` (unchanged)
5. `_build_research_tools()` (renamed from `_build_tools`)
6. `_web_research_agent = create_react_agent(...)`
7. `@tool async def web_research(...)` — wraps `_web_research_agent`
8. `_root_agent = create_react_agent(..., tools=[web_research])`
9. `async def get_response(...)` (unchanged signature)

## Key Design Decisions

| Decision | Rationale |
|---|---|
| Agent-as-tool pattern | Simplest LangGraph multi-agent approach; no new dependencies |
| `async def web_research` | Must be async to call `_web_research_agent.ainvoke()` |
| Root agent has NO direct web tools | Forces all internet access through the subagent boundary |
| Sub-agent tools remain sync | Unchanged from v2.0.0; LangGraph runs sync tools in thread executor |
| Error returns string (not raise) | Raising inside `@tool` causes LangGraph to loop; string lets root agent relay gracefully |
| Shared `_llm` instance | Both agents use the same GPT-4o model |
| Stateless sub-agent calls | Each `web_research` invocation is independent; root agent composes full task description |
